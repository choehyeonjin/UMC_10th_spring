- 오케스트레이션 서비스와 쿠버네티스

  참고 자료: https://blog.naver.com/brainzsquare/223342037399

    - 컨테이너 오케스트레이션
        - 여러 컨테이너에 대한 프로세스를 최적화하고 적절한 자원의 할당과 자동으로 컨테이너를 생성하고 배포하는 일
        - 마이크로서비스 아키텍처의 각각의 서비스를 구현하는 데에 사용된다.
    - 쿠버네티스

      ![image.png](./keyword_05_appendix_(1).jpg)

        - 컨테이너화된 어플리케이션을 자동으로 배포하고 스케일링 및 관리하기 위한 컨테이너 오케스트레이션 도구
        - 쿠버네티스에서 지원하는 오케스트레이션 서비스는 다음과 같다.
        1. **오토스케일링 (Auto-Scaling)**

           쿠버네티스에서 생성하고 관리할 수 있는 가장 작은 컴퓨팅 단위를 파드(Pod)라고 한다. 만약 부하가 증가하여 리소스를 과하게 점유하고 있다면 자동으로 파드 복제본이 실행되어 가용성을 확보한다.

        2. **스케줄링 (Scheduling)**

           컨테이너를 일정한 알고리즘에 기초하여 구체적으로 어떤 노드에서 움직이게 할지 배치치한다. ‘Kube Scheduler’라는 컴포넌트를 통해 클러스터 내에 실행할 파드를 노드에 스케줄링한다.

        3. **오토 힐링 (Auto-Healing)**

           컨테이너의 상태를 지속적으로 관찰하여 비정상적인 상태를 감지하면 컨테이너를 재시작하고 스케줄링을 빠르게 재시작한다.

        4. **분산 부하 (Load-Balancing)**

           하나의 서비스에 여러 개의 컨테이너가 구동 시, 서비스에 들어오는 요청을 컨테이너들 사이에 균등하게 분배하여 부하를 분산시킨다.

- EDA

  참고 자료: https://aws.amazon.com/ko/what-is/eda/

    - Event-driven architecture, 이벤트 기반 아키텍처(EDA)는 이벤트를 게시, 소비 또는 라우팅하는 분리된 소형 서비스에서 구축된 현대적 아키텍처 패턴이다.
    - 동작 흐름
        1. Producer → 이벤트 생성
        2. Router → 이벤트 전달/필터링
        3. Consumer → 이벤트 처리

      → 서비스들이 **서로 직접 호출하지 않고 이벤트로 통신함**

      예시:

        - 주문 생성 → 이벤트 발생
        - 결제 / 배송 / 재고 서비스가 각각 처리
    - 핵심 특징
        - 특정 서비스에 장애가 발생하더라도 나머지는 계속 실행된다. 실시간에 가까운 시스템을 더 쉽게 설계할 수 있다.
        - 이벤트 라우터는 애플리케이션을 감사하고 정책을 정의하는 중앙 집중식 로케이션 역할을 한다.
        - 이벤트 기반 아키텍처는 푸시 기반이다. 즉, 종속 서비스에 알릴 필요 없이 이벤트가 라우터 및 다운스트림 시스템으로 전송될 때 모든 작업이 온디맨드 방식으로 수행된다.
        - 확장성이 높다. 이벤트를 게시함으로써 애플리케이션을 기존 시스템과 통합할 수 있으며, 향후 기존 솔루션을 변경하지 않고도 애플리케이션을 이벤트 소비자로서 손쉽게 통합할 수 있습니다.
- 카프카

  참고 자료: https://medium.com/@greg.shiny82/apache-kafka-%EA%B0%84%EB%9E%B5%ED%95%98%EA%B2%8C-%EC%82%B4%ED%8E%B4%EB%B3%B4%EA%B8%B0-343ad84a959b

    - Apache Kafka는 실시간으로 스트리밍 데이터를 수집하고 처리하는 데 최적화된 분산 데이터 저장소 오픈소스이다. 다음은 카프카의 구성 요소이다.
    - **브로커(Broker)**
        - 하나의 카프카 서버를 브로커라고 한다.
        - 브로커는 프로듀서로부터 메시지를 수신하고 오프셋을 지정한 후 해당 메시지를 디스크에 저장한다.
        - 또한 컨슈머의 파티션 읽기 요청에 응답하고 디스크에 수록된 메시지를 전송한다.
        - 여러 개의 브로커가 하나의 클러스터에 포함될 수 있으며, 그 중 하나는 클러스터의 컨트롤러(Controller) 역할을 수행한다.
        - 컨트롤러는 클러스터 내의 각 브로커에게 담당 파티션을 할당하고, 브로커들이 정상적으로 동작하는지 모니터링한다.
    - **클러스터 (Cluster)**
        - 여러 대의 브로커가 모인 하나의 거대한 서비스를 클러스터라고 한다.
        - 특정 서버에 장애가 발생하더라도 카프카를 이용하는 클라이언트에게는 정상적인 처리와 응답을 제공할 수 있다.
    - **토픽과 파티션 (Topic, Partition)**
        - 카프카에서 메시지는 데이터의 기본 단위이며, 토픽(Topic)으로 분리된다. 하나의 토픽은 여러 개의 파티션(Partition) 으로 구성된다.
        - 이 때 각 파티션은 서로 다른 서버에 분산될 수 있는데, 이러한 특징 때문에 하나의 토픽이 여러 서버에 걸쳐 수평적으로 확장될 수 있다.
    - **프로듀서와 컨슈머 (Producer, Consumer)**
        - 카프카의 클라이언트는 프로듀서(Producer) 와 컨슈머(Consumer) 가 있다.
        - 프로듀서는 새로운 메시지를 특정 토픽에 생성하는데, 메시지 키와 파티셔너를 활용한다.
        - 컨슈머는 하나 이상의 토픽을 구독하면서 메시지가 생성된 순서로 읽는다.
    - **컨슈머 그룹**
        - 카프카 컨슈머들은 컨슈머 그룹(Consumer Group) 에 속하게 된다. 여러 개의 컨슈머가 같은 컨슈머 그룹에 속할 때에는 각 컨슈머가 해당 토픽의 다른 파티션을 분담해서 메시지를 읽을 수 있다.
        - 이처럼 하나의 컨슈머 그룹에 더 많은 컨슈머를 추가하면 카프카 토픽의 데이터 소비를 확장할 수 있다.
        - 다중 컨슈머 기능이란 여러 개의 컨슈머 그룹이 서로간의 상호 간섭 없이 각자의 오프셋으로 각자의 순서에 맞게 메시지를 읽고 처리할 수 있는 것을 말한다.
- 메시지큐와 이벤트 소싱

  참고 자료: https://yoonseon.tistory.com/173

    - 메시지 큐
        - 메시지를 발행하는 생산자가 있고, 메시지를 수신하는 소비자가 있다. 메시지는 큐에 쌓이고 소비자가 읽으면 큐에서 사라진다.
        - 특징:
            - 비동기 처리: 시스템 간 병목현상 제거 및 응답 속도 향상.
            - 메시지 내구성: 큐에 있는 메시지는 성공적으로 소비될 때까지 저장됨.
            - 부하 분산: 여러 소비자가 있으면 메시지가 분산됨.
    - 이벤트 소싱
        - 시스템에서 발생한 모든 사실을, 발생 순서대로 불변 이벤트로 저장하는 방식이다.
        - 정보를 잃지 않으면서 사용자의 행동 흐름을 쉽게 추론할 수 있고, 필요하다면 과거 상태를 재구성하거나 되돌리는 것을 가능하게 하는 새로운 모델링 패러다임이다.

- Spring Cloud Eureka

  참고 자료: https://baebalja.tistory.com/612

    - Spring Cloud Eureka는 스프링 프레임워크에서 마이크로서비스 아키텍처 구현을 위해 제공하는 기술이다.
    - 인스턴스의 목록과 위치(Host, Port)가 동적으로 변하는 MSA 환경에서 사용자가 각 서비스의 위치를 모두 관리하기가 어렵다.
      Eureka는 중간 계층 서버의 로드 밸런싱 및 장애 조치를 목적으로 서비스를 찾기 위해 클라우드에서 주로 사용되는 REST(Representational State Transfer) 기반 서비스이다.
    - 클라이언트가 요청을 보내면 API Gateway에 전달되고 Eureka 서버에 가서 필요한 서비스가 어느 곳에 있는지 찾고 해당 서비스로 호출하게 된다.
