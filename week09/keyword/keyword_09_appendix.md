- FIDO란?

  **FIDO**는 **Fast IDentity Online**의 약자로, 비밀번호 대신 **공개키 암호 방식**을 이용해서 안전하게 로그인하는 인증 표준이다.

    ```
    비밀번호를 서버에 보내서 로그인하는 방식이 아니라,
    내 기기 안에 있는 개인키로 인증하고
    서버는 공개키로 그 결과가 맞는지만 확인하는 방식
    ```

  ## 기존 비밀번호 로그인

    ```
    사용자 → 아이디/비밀번호 입력
    서버 → 비밀번호가 맞는지 확인
    ```

  문제점:

    ```
    비밀번호 유출 가능
    피싱 사이트에 입력할 위험
    여러 사이트에서 같은 비밀번호 재사용
    서버 DB 털리면 해시된 비밀번호도 위험
    ```

  ## FIDO 로그인

    ```
    사용자 기기 → 개인키 보관
    서버 → 공개키 보관
    로그인 시 → 서버가 challenge를 줌
    사용자 기기 → 개인키로 challenge에 서명
    서버 → 공개키로 서명 검증
    ```

  즉 서버에는 **비밀번호가 아니라 공개키만 저장**됨.

  ## 핵심 구성

  ### 1. 개인키

  사용자의 기기 안에 저장됨.

  예:

    ```
    스마트폰
    노트북
    보안키
    ```

  외부로 잘 빠져나가지 않음.

  ### 2. 공개키

  서버 DB에 저장됨.

  공개키는 말 그대로 공개되어도 되는 키라서, 비밀번호보다 서버 유출 피해가 훨씬 적음.

  ### 3. Challenge

  서버가 매번 새로 만드는 랜덤값.

    ```
    서버: 이 랜덤값에 서명해서 보내라
    기기: 개인키로 서명해서 보냄
    서버: 공개키로 검증
    ```

  재사용 공격을 막기 위한 값임.

  ## FIDO와 패스키 관계

    ```
    FIDO = 인증 표준/기술 규격
    WebAuthn = 웹에서 FIDO를 쓰기 위한 브라우저 API 표준
    Passkey = 사용자가 체감하는 비밀번호 없는 로그인 방식
    ```

  정리하면:

    ```
    FIDO라는 기술 표준을
    웹에서는 WebAuthn으로 구현하고
    사용자는 패스키라는 형태로 사용한다
    ```

  ## FIDO의 장점

    ```
    비밀번호를 서버에 저장하지 않음
    피싱에 강함
    재사용 가능한 비밀번호가 없음
    생체인증/PIN으로 편하게 로그인 가능
    서버 DB가 털려도 개인키는 유출되지 않음
    ```

- challenge-response authentication

  **Challenge-Response Authentication**은 서버가 낸 문제를 사용자가 가진 비밀 정보로 풀어서, 진짜 사용자인지 증명하는 인증 방식이다.

  ## 1. 기본 개념

    ```
    Challenge = 서버가 보내는 랜덤 문제값
    Response = 사용자가 그 문제값을 처리해서 보내는 답
    ```

    ```
    서버: 이 랜덤값을 네 비밀키로 처리해서 보내라
    사용자: 처리한 결과 보냄
    서버: 결과가 맞는지 검증
    ```

  여기서 중요한 점은 **비밀번호나 개인키 자체를 서버로 보내지 않는다**는 것이다.

  ## 2. 왜 쓰는가?

  기존 로그인은 보통 이렇게 함.

    ```
    사용자 → 비밀번호 전송
    서버 → 비밀번호 확인
    ```

  이 방식은 비밀번호가 네트워크 중간에서 탈취되거나, 피싱 사이트에 입력될 위험이 있음.

  반면 Challenge-Response 방식은:

    ```
    비밀값 자체는 보내지 않음
    매번 다른 랜덤값에 대한 응답만 보냄
    ```

  그래서 누가 이전 응답값을 훔쳐도 다음 로그인에 재사용하기 어려움.

  ## 3. 동작 흐름

  예를 들어 패스키/WebAuthn에서는 이렇게 동작함.

    ```
    1. 클라이언트가 로그인 요청
    2. 서버가 challenge라는 랜덤값 생성
    3. 서버가 challenge를 클라이언트에게 전달
    4. 사용자의 기기가 개인키로 challenge에 서명
    5. 클라이언트가 서명 결과를 서버에 전달
    6. 서버는 저장된 공개키로 서명을 검증
    7. 검증 성공 시 로그인 성공
    ```

  ## 4. WebAuthn에서의 예시

  서버가 옵션 요청에서 challenge를 내려줌.

    ```json
    {
      "challenge":"random-value-123",
      "rpId":"example.com",
      "allowCredentials": [
        {
          "type":"public-key",
          "id":"credential-id"
        }
      ]
    }
    ```

  브라우저는 이 challenge를 가지고 인증을 진행함.

    ```jsx
    navigator.credentials.get({
      publicKey:options
    });
    ```

  사용자의 기기는 내부에 있는 개인키로 challenge에 서명함.

  그리고 서버는 응답으로 온 서명을 검증함.

    ```
    서버에 저장된 공개키로 검증 가능 → 진짜 사용자
    검증 실패 → 가짜 사용자 또는 변조된 요청
    ```

  ## 5. 핵심 요소

  ### Challenge

  서버가 만든 **일회용 랜덤값**.

    ```
    매번 달라야 함
    예측 불가능해야 함
    재사용되면 안 됨
    ```

  이게 있어야 재전송 공격을 막을 수 있음.

  ### Response

  challenge를 사용자의 비밀 정보로 처리한 결과.

  패스키에서는 보통:

    ```
    개인키로 challenge에 전자서명한 값
    ```

  ### 검증 정보

  서버가 response가 맞는지 확인하는 데 쓰는 정보.

  패스키에서는:

    ```
    서버 DB에 저장된 공개키
    ```

  를 사용함.

  ## 6. Replay Attack 방지

  Replay Attack은 공격자가 이전 로그인 응답을 훔쳐서 그대로 다시 보내는 공격이다.

  예를 들어 challenge가 항상 같다면:

    ```
    공격자: 전에 훔친 response 다시 보내야지
    서버: 맞네? 로그인 성공
    ```

  이렇게 될 수 있음.

  하지만 challenge가 매번 다르면:

    ```
    이전 response는 이전 challenge에 대한 답
    현재 challenge에는 맞지 않음
    ```

  그래서 재사용이 안 됨.

  ## 7. 비밀번호 로그인과 비교

  | 구분 | 비밀번호 로그인 | Challenge-Response |
      | --- | --- | --- |
  | 서버로 보내는 것 | 비밀번호 | challenge에 대한 응답 |
  | 비밀값 전송 여부 | 전송됨 | 전송되지 않음 |
  | 재사용 공격 위험 | 상대적으로 높음 | 낮음 |
  | 대표 예시 | ID/PW 로그인 | FIDO, WebAuthn, 패스키 |
  | 서버 저장 정보 | 비밀번호 해시 | 공개키 등 검증 정보 |

  ## 8. 패스키에서의 의미

  패스키에서는 이 구조가 핵심이다.

    ```
    서버는 challenge를 줌
    기기는 개인키로 challenge에 서명함
    서버는 공개키로 서명을 검증함
    ```

  즉 패스키 로그인은 결국:

    ```
    Challenge-Response Authentication 기반의 공개키 인증 방식
    ```

  이라고 볼 수 있음.

- 왜 비대칭키 암호화 방식이 패스키에 적절한지

  패스키(WebAuthn/FIDO)는 기본적으로 **비대칭키 암호화(Public Key Cryptography)** 를 기반으로 동작한다.

  비대칭키 방식은:

    ```
    개인키(Private Key) + 공개키(Public Key)
    ```

  한 쌍의 키를 사용하는 방식이다.

  패스키에서는:

    ```
    개인키 → 사용자 기기 내부에만 저장
    공개키 → 서버 DB에 저장
    ```

  된다.
    
  ---

  # 1. 비밀번호를 서버에 저장하지 않아도 됨

  기존 비밀번호 로그인은 서버가 사용자의 비밀번호(또는 해시)를 저장해야 한다.

    ```
    사용자 → 비밀번호 입력
    서버 → 저장된 비밀번호 해시와 비교
    ```

  문제는 서버 DB가 유출되면 비밀번호 해시도 함께 위험해진다는 점이다.

  반면 패스키는:

    ```
    서버에는 공개키만 저장
    개인키는 사용자 기기 밖으로 절대 나오지 않음
    ```

  즉 서버가 털려도 공격자는 공개키만 얻는다.

  공개키만으로는 로그인할 수 없기 때문에 훨씬 안전하다.
    
  ---

  # 2. 개인키를 서버로 전송하지 않음

  패스키 인증에서는:

    ```
    서버 → challenge 전송
    기기 → 개인키로 challenge 서명
    서버 → 공개키로 검증
    ```

  을 수행한다.

  여기서 중요한 점은:

    ```
    개인키 자체는 네트워크로 절대 전송되지 않는다
    ```

  는 것이다.

  따라서:

    ```
    중간자 공격
    패킷 탈취
    네트워크 감청
    ```

  등이 발생해도 개인키 유출 위험이 매우 낮다.
    
  ---

  # 3. 피싱 공격에 강함

  기존 비밀번호 방식은 사용자가 직접 비밀번호를 입력한다.

  따라서 가짜 사이트에 속아 비밀번호를 입력하면 그대로 탈취된다.

  하지만 패스키는:

    ```
    특정 도메인(origin)에 연결된 공개키 기반 인증
    ```

  을 사용한다.

  예를 들어:

    ```
    정상 사이트:
    https://google.com
    
    피싱 사이트:
    https://goog1e.com
    ```

  이라면 브라우저는 origin이 다르다고 판단한다.

  그래서:

    ```
    google.com용 패스키는
    goog1e.com에서는 사용되지 않음
    ```

  즉 사용자가 속더라도 개인키 서명이 수행되지 않아 피싱에 매우 강하다.
    
  ---

  # 4. Replay Attack 방지 가능

  패스키는 매 인증마다 서버가 새로운 challenge를 생성한다.

    ```
    challenge = 일회용 랜덤값
    ```

  기기는 개인키로 challenge에 서명한다.

  따라서 이전 응답을 탈취해도:

    ```
    이전 challenge에 대한 응답일 뿐
    새 challenge에는 사용할 수 없음
    ```

  즉 재전송 공격(Replay Attack)을 방지할 수 있다.
    
  ---

  # 5. 서버가 사용자 비밀을 알 필요가 없음

  대칭키 방식에서는 서버와 사용자가 같은 비밀키를 공유해야 한다.

  하지만 비대칭키 방식에서는:

    ```
    사용자만 개인키 보유
    서버는 공개키만 보유
    ```

  하면 된다.

  즉 서버는 사용자의 실제 비밀값(개인키)을 전혀 몰라도 인증이 가능하다.

  이 구조가 패스키 보안의 핵심이다.
    
  ---

  # 6. 사용자 편의성이 높음

  개인키는 기기 내부 안전 영역에 저장되고:

    ```
    지문
    얼굴인식
    PIN
    ```

  등으로 잠금 해제만 하면 된다.

  즉 사용자는 복잡한 비밀번호를 기억할 필요가 없다.

  하지만 내부적으로는 강력한 공개키 암호 기반 인증이 수행된다.
    
  ---

  # 대칭키 방식과 비교

  | 구분 | 대칭키 방식 | 비대칭키 방식(패스키) |
      | --- | --- | --- |
  | 키 개수 | 하나 | 공개키 + 개인키 |
  | 서버 저장 정보 | 비밀키 | 공개키 |
  | 서버 유출 위험 | 큼 | 상대적으로 낮음 |
  | 개인 비밀 전송 여부 | 가능성 있음 | 없음 |
  | 피싱 대응 | 약함 | 강함 |
  | 패스키 적합성 | 낮음 | 매우 높음 |

  # 정리

  패스키는:

    ```
    개인키는 사용자 기기에만 저장
    서버는 공개키만 저장
    개인키는 외부로 전송되지 않음
    challenge 기반 서명 검증 수행
    ```

  이라는 구조를 사용한다.

  따라서 비대칭키 암호화 방식은:

    ```
    서버 유출 대응
    피싱 방지
    재전송 공격 방지
    개인키 보호
    ```

  등의 장점을 제공하므로 패스키 인증에 매우 적합하다.
