- 세션과 토큰의 차이는?

  세션과 토큰은 둘 다 사용자의 로그인 상태를 유지하기 위해 사용된다.

  하지만 인증 정보를 어디에 저장하느냐가 가장 큰 차이다.
    
  ---

  ## 1. 세션 방식

  세션 방식은 **서버**가 사용자의 로그인 정보를 저장하는 방식이다.

  사용자가 로그인하면 서버는 해당 사용자에 대한 **세션**을 만들고, **세션 ID**를 클라이언트에게 전달한다.

  클라이언트는 이 세션 ID를 **쿠키**에 저장하고, 이후 요청마다 서버에 함께 보낸다.

  서버는 전달받은 세션 ID를 보고 **서버에 저장된 세션 정보**를 찾아 사용자를 인증한다.

  ### 흐름

    1. 사용자가 로그인한다.
    2. 서버가 세션을 생성한다.
    3. 서버는 세션 ID를 클라이언트에게 보낸다.
    4. 클라이언트는 세션 ID를 쿠키에 저장한다.
    5. 이후 요청마다 쿠키에 담긴 세션 ID를 서버로 보낸다.
    6. 서버는 세션 저장소에서 세션 ID를 조회해 사용자를 확인한다.

    ---

  ## 2. 토큰 방식

  토큰 방식은 서버가 로그인 정보를 저장하지 않고, **클라이언트**가 토큰을 가지고 있는 방식이다.

  사용자가 로그인하면 서버는 사용자 정보를 담은 토큰을 발급한다.

  클라이언트는 이 토큰을 저장해두고, 이후 요청마다 `Authorization` 헤더에 담아 서버로 보낸다.

  서버는 토큰의 **서명과 만료 시간을 검증**해서 사용자를 인증한다.

  대표적인 토큰 방식으로는 JWT가 있다.

  ### 흐름

    1. 사용자가 로그인한다.
    2. 서버가 토큰을 발급한다.
    3. 클라이언트는 토큰을 저장한다.
    4. 이후 요청마다 토큰을 서버에 보낸다.
    5. 서버는 토큰이 유효한지 검증한다.
    6. 유효하면 토큰 안의 사용자 정보를 바탕으로 인증한다.

    ---

  ## 핵심 차이

  | 구분 | 세션 | 토큰 |
      | --- | --- | --- |
  | 인증 정보 저장 위치 | 서버 | 클라이언트 |
  | 서버 상태 | Stateful | Stateless |
  | 확장성 | 서버 간 세션 공유 필요 | 서버 확장에 유리 |
  | 인증 확인 방식 | 세션 저장소 조회 | 토큰 자체 검증 |
  | 로그아웃 처리 | 서버 세션 삭제 | 토큰 만료 전까지 유효할 수 있음 |
  | 보안 관리 | 서버에서 세션 제어 쉬움 | 토큰 탈취 시 위험 |
  | 주 사용 위치 | 전통적인 웹 서비스 | REST API, 모바일, SPA |
    
  ---

  ## 3. Stateful vs Stateless 관점

  세션 방식은 **Stateful**하다.

  서버가 사용자의 로그인 상태를 기억하고 있기 때문이다.

  반면 토큰 방식은 **Stateless**하다.

  서버가 로그인 상태를 저장하지 않고, 요청에 포함된 토큰만 보고 인증하기 때문이다.

  즉, 세션은 서버가 “이 사용자는 로그인한 상태야”라고 기억하는 방식이고,

  토큰은 클라이언트가 “나는 인증된 사용자야”라는 증명서를 들고 오는 방식이다.
    
  ---

  ## 4. 장단점

  ### 세션 장점

  서버가 로그인 상태를 직접 관리하므로 제어가 쉽다.

  예를 들어 로그아웃하면 서버에서 세션을 삭제하면 되고, 강제 로그아웃도 쉽게 처리할 수 있다.

  ### 세션 단점

  서버가 세션 정보를 저장해야 하므로 사용자가 많아지면 부담이 커진다.

  서버가 여러 대라면 세션을 공유하기 위해 Redis 같은 별도의 세션 저장소가 필요할 수 있다.
    
  ---

  ### 토큰 장점

  서버가 세션을 저장하지 않아도 되므로 확장성이 좋다.

  여러 서버가 있어도 같은 비밀키나 공개키로 토큰을 검증할 수 있기 때문에 분산 환경에 적합하다.

  ### 토큰 단점

  한 번 발급된 토큰은 만료되기 전까지 유효할 수 있다.

  그래서 로그아웃이나 강제 차단을 구현하려면 블랙리스트, 짧은 만료 시간, Refresh Token 같은 추가 설계가 필요하다.

- 엑세스 토큰과 리프레시 토큰이란?

  ## 1. 액세스 토큰

  액세스 토큰은 사용자가 API에 접근할 때 사용하는 토큰이다.

  로그인 후 서버가 액세스 토큰을 발급하면, 클라이언트는 이후 요청마다 이 토큰을 서버에 보낸다.

  보통 요청 헤더에 담아 보낸다.

    ```json
    Authorization: Bearer {액세스토큰}
    ```

  서버는 액세스 토큰을 검증해서 사용자가 누구인지 확인하고, 요청한 기능을 사용할 수 있는지 판단한다.

  ### 특징

  | 구분 | 내용 |
      | --- | --- |
  | 목적 | API 접근 권한 증명 |
  | 사용 위치 | 매 요청마다 서버에 전달 |
  | 유효 시간 | 짧게 설정 |
  | 탈취 위험 | 탈취되면 바로 API 접근 가능 |
  | 예시 | 게시글 작성, 내 정보 조회, 주문 요청 |
    
  ---

  ## 2. 리프레시 토큰

  리프레시 토큰은 액세스 토큰을 다시 발급받기 위한 토큰이다.

  액세스 토큰은 보안을 위해 유효 시간을 짧게 가져간다.

  하지만 액세스 토큰이 만료될 때마다 사용자가 다시 로그인해야 하면 불편하다.

  그래서 리프레시 토큰을 사용한다.

  액세스 토큰이 만료되면 클라이언트는 리프레시 토큰을 서버에 보내고, 서버는 리프레시 토큰이 유효한지 확인한 뒤 새로운 액세스 토큰을 발급한다.

  ### 특징

  | 구분 | 내용 |
      | --- | --- |
  | 목적 | 액세스 토큰 재발급 |
  | 사용 위치 | 액세스 토큰 만료 시 사용 |
  | 유효 시간 | 길게 설정 |
  | 탈취 위험 | 탈취되면 계속 토큰 재발급 가능 |
  | 예시 | 자동 로그인 유지 |
    
  ---

  ## 3. 왜 두 개로 나누는가?

  토큰을 하나만 사용하면 문제가 있다.

  유효 시간을 길게 설정하면 사용자는 편하지만, 토큰이 탈취됐을 때 오랫동안 악용될 수 있다.

  반대로 유효 시간을 짧게 설정하면 보안은 좋아지지만, 사용자가 자주 다시 로그인해야 한다.

  그래서 역할을 나눈다.

  액세스 토큰은 짧게 가져가서 API 요청에 사용하고,

  리프레시 토큰은 길게 가져가서 액세스 토큰을 재발급하는 데 사용한다.
    
  ---

  ## 4. 동작 흐름

    1. 사용자가 로그인한다.
    2. 서버가 액세스 토큰과 리프레시 토큰을 발급한다.
    3. 클라이언트는 액세스 토큰으로 API를 요청한다.
    4. 서버는 액세스 토큰을 검증하고 요청을 처리한다.
    5. 액세스 토큰이 만료된다.
    6. 클라이언트는 리프레시 토큰으로 재발급을 요청한다.
    7. 서버는 리프레시 토큰을 검증한다.
    8. 유효하면 새로운 액세스 토큰을 발급한다.
    9. 리프레시 토큰도 만료되었거나 유효하지 않으면 다시 로그인해야 한다.

    ---

  ## 5. 핵심 차이

  | 구분 | 액세스 토큰 | 리프레시 토큰 |
      | --- | --- | --- |
  | 역할 | API 접근 | 액세스 토큰 재발급 |
  | 유효 시간 | 짧음 | 김 |
  | 서버로 보내는 빈도 | 매 요청마다 | 액세스 토큰 만료 시 |
  | 탈취 시 위험 | 만료 전까지 API 사용 가능 | 새로운 액세스 토큰 계속 발급 가능 |
  | 저장 위치 | 메모리, 쿠키 등 | 쿠키 또는 보안이 강한 DB/Redis |
  | 만료 시 처리 | 리프레시 토큰으로 재발급 | 다시 로그인 |
    
  ---

  ## 6. 보안 관점

  액세스 토큰은 자주 사용되기 때문에 탈취 가능성을 줄이기 위해 유효 시간을 짧게 설정한다.

  리프레시 토큰은 오래 살아있기 때문에 더 안전하게 관리해야 한다.

  보통 서버 DB나 Redis에 저장해두고, 로그아웃 시 삭제하거나 만료 처리한다.

  또한 리프레시 토큰을 사용할 때마다 새 리프레시 토큰을 다시 발급하는 방식도 있다.

  이를 Refresh Token Rotation이라고 한다.

  이 방식을 사용하면 기존 리프레시 토큰이 탈취되었을 때 재사용을 감지하거나 차단하기 쉽다.

- OAuth 1.0과 OAuth 2.0의 차이는?

  # OAuth 1.0과 OAuth 2.0의 차이

  OAuth는 사용자의 아이디와 비밀번호를 직접 전달하지 않고,

  제3자 애플리케이션이 제한된 권한으로 사용자 자원에 접근할 수 있도록 하는 인증/인가 프로토콜이다.

  대표적으로 “카카오 로그인”, “구글 로그인” 같은 소셜 로그인이 OAuth 기반이다.
    
  ---

  # 1. OAuth 1.0

  OAuth 1.0은 초기 OAuth 표준이다.

  보안 강화를 위해 요청마다 복잡한 서명(Signature) 과정을 사용했다.

  클라이언트는 요청을 보낼 때 다음과 같은 정보를 이용해 서명을 생성한다.

    - Consumer Secret
    - Token Secret
    - Timestamp
    - Nonce
    - 요청 URL
    - HTTP Method

  서버는 동일한 방식으로 서명을 계산해 요청 위변조 여부를 검증한다.
    
  ---

  ## 특징

  | 구분 | 내용 |
      | --- | --- |
  | 보안 방식 | 요청마다 Signature 생성 |
  | 토큰 | Access Token |
  | HTTPS 의존성 | 낮음 |
  | 구현 난이도 | 높음 |
  | 사용 편의성 | 복잡 |
  | 현재 사용 | 거의 없음 |
    
  ---

  ## 문제점

  OAuth 1.0은 보안은 강했지만 구조가 매우 복잡했다.

    - Signature 생성 과정이 어려움
    - 구현 실수 가능성 높음
    - 개발 생산성 낮음
    - 모바일/SPA 환경 대응 어려움

  그래서 더 단순한 OAuth 2.0이 등장했다.
    
  ---

  # 2. OAuth 2.0

  OAuth 2.0은 OAuth 1.0을 단순화하고 확장성을 높인 버전이다.

  가장 큰 특징은 복잡한 Signature 제거 + HTTPS 기반 보안 사용

  즉, OAuth 1.0처럼 요청마다 서명하지 않고 HTTPS 암호화 통신 자체를 신뢰한다.
    
  ---

  ## 특징

  | 구분 | 내용 |
      | --- | --- |
  | 보안 방식 | HTTPS 기반 |
  | 토큰 | Access Token + Refresh Token |
  | 구현 난이도 | 낮음 |
  | 사용 편의성 | 높음 |
  | 모바일/SPA 지원 | 좋음 |
  | 현재 사용 | 대부분 OAuth 2.0 |
    
  ---

  # 3. OAuth 2.0의 주요 구성 요소

  | 구성 요소 | 설명 |
      | --- | --- |
  | Resource Owner | 사용자 |
  | Client | 로그인 기능 사용하는 앱 |
  | Authorization Server | 인증 서버 |
  | Resource Server | 사용자 정보 제공 서버 |
    
  ---

  # 4. OAuth 2.0 인증 흐름

  예: 구글 로그인

    1. 사용자가 “구글 로그인” 클릭
    2. 클라이언트가 구글 인증 서버로 이동
    3. 사용자가 로그인 및 권한 동의
    4. 구글이 Authorization Code 발급
    5. 클라이언트가 Code로 Access Token 요청
    6. 구글이 Access Token 발급
    7. 클라이언트가 토큰으로 사용자 정보 요청

    ---

  # 5. 핵심 차이

  | 구분 | OAuth 1.0 | OAuth 2.0 |
      | --- | --- | --- |
  | 보안 방식 | Signature 기반 | HTTPS 기반 |
  | 요청 검증 | 요청마다 서명 | 토큰 검증 |
  | 구현 난이도 | 복잡 | 단순 |
  | 성능 | 상대적으로 느림 | 상대적으로 빠름 |
  | 모바일 지원 | 부족 | 우수 |
  | Refresh Token | 없음 | 있음 |
  | 확장성 | 낮음 | 높음 |
  | 현재 사용률 | 거의 없음 | 표준 수준 |
    
  ---

  # 6. Refresh Token 지원 차이

  OAuth 1.0은 일반적으로 Access Token만 사용했다.

  OAuth 2.0은:

    - Access Token
    - Refresh Token

  구조를 지원한다.

  그래서 사용자는 로그인 상태를 유지하면서도 액세스 토큰을 짧게 운영할 수 있다.
    
  ---

  # 7. 왜 OAuth 2.0이 많이 사용되는가?

  OAuth 2.0은:

    - 구현이 단순함
    - 모바일 앱 지원 쉬움
    - SPA 지원 쉬움
    - API 중심 구조에 적합
    - 토큰 기반 인증과 잘 맞음

  그래서 현재 대부분의 소셜 로그인은 OAuth 2.0 기반이다.
